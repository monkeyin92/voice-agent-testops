# 开源边界与商业护城河

更新日期：2026-05-07

项目开源不等于护城河消失。相反，早期把基础能力、示例和方法论开源，可以降低试用门槛，建立可信度，并让开发者愿意把自己的 voice agent 接进来。真正需要保护的是会随客户使用持续复利的数据、判断和工作流。

## 可以开源的部分

这些内容适合放在公开仓库里：

- 通用 CLI、schema、runner、reporter、adapter 示例。
- starter suite 和少量原创 seed annotation。
- 行业风险 rubric 的基础版本。
- mock data 指南、接入指南、报告示例。
- 用公开信息整理出的数据源调研和标注方法。
- 可复现的 demo 和失败样例。

开源这些内容的价值是让别人快速理解产品形态：我们不是卖“几条正则”，而是把 voice agent 上线风险变成可重复的测试门禁。

## 不应该开源的部分

这些内容应留在客户项目、私有数据仓库或商业版服务里：

- 客户真实通话、录音、转写和脱敏前原始数据。
- 客户专属事实表、价格政策、知识库、内部 SOP。
- 客户专属 regression suite 和上线阻断历史。
- 人工复核后的高质量标签、分歧记录和 adjudication 结论。
- 每个行业的误报/漏报分析、judge 校准结果和阈值。
- 生产监控抽样、失败聚类、客户版本间 baseline diff。
- 真实 adapter 配置、认证信息、webhook、CRM/工单/呼叫中心集成细节。
- 销售复盘、客户 ROI、行业 benchmark 和竞品对比。

这些才是会随着客户数量和行业深度越来越厚的资产。公开仓库只能看到“怎么开始”，看不到“某个真实客户上线前最怕什么、过去失败过什么、下一次应该挡住什么”。

## 为什么公开 seed 不会削弱护城河

公开 seed set 的定位是样板间，不是生产数据库。它有三个作用：

1. 证明产品判断标准。

   用户可以直接看到哪些场景会被判为风险，为什么需要 `semantic_judge`，以及规则断言和语义断言如何配合。

2. 降低接入成本。

   开发者不需要从空白 JSON 开始，可以先跑通三行业 starter，再把自己的真实通话转成 suite。

3. 形成事实上的格式标准。

   当别人按我们的 schema 写 suite、按我们的报告看失败、按我们的 rubric 讨论风险时，项目的开源部分会变成分发渠道。

真正的差距不在“有没有 45 条样例”，而在：

- 能不能从客户真实通话中持续发现新失败。
- 能不能把失败转成稳定 regression suite。
- 能不能让业务、客服、合规、工程都认可同一个 failure reason。
- 能不能在上线、监控、回归、复盘之间形成闭环。

## 建议的商业分层

开源版：

- CLI、本地 runner、基础报告。
- 通用 starter 和少量行业 seed。
- HTTP/OpenClaw/Vapi/Retell 等最短接入示例。
- 适合开发者自助试用和技术传播。

商业版：

- 私有行业包和客户专属 suite 托管。
- LLM judge 校准、人工复核、误报/漏报分析。
- 生产通话抽样监控和失败聚类。
- baseline diff 上线门禁。
- 企业 adapter、权限、审计、报表和客户成功复盘。

服务/交付：

- 帮客户从历史通话和投诉记录中建立第一批 regression suite。
- 每周/每月复盘新增风险和误报。
- 和客户业务 owner 一起维护风险口径。

## 仓库维护规则

1. 公共 examples 只能使用原创、合成或明确授权的数据。
2. 公开数据集只能作为标签体系、intent、slot 和标注字段参考，不直接复制原始对话。
3. 客户真实 transcript 默认不进公开仓库，即使脱敏后也要单独确认授权。
4. 如果某个样本来自客户失败案例，只能抽象成行业化合成样本，删除客户名、价格、地点、话术习惯和特殊政策。
5. 每新增一个公开 seed，都要能说明它保护的通用风险，而不是暴露某个客户的私有业务。

## 对外表达

可以这样解释：

> 开源仓库给你看的是测试框架、样例和方法论；商业价值在持续帮你的 voice agent 吸收真实失败、沉淀行业风险、校准语义评测，并把这些风险变成上线门禁和生产监控。

这也是为什么公开一部分数据不是坏事。它让市场知道我们在测什么，但不会替代我们和客户一起积累的私有失败闭环。
