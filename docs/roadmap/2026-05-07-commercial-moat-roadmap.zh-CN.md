# Voice Agent TestOps 商业化护城河路线图

日期：2026-05-07

## 0. 北极星

Voice Agent TestOps 不做“敏感词扫描器”，也不做通用聊天机器人评分器。

它的北极星是：

> 给 voice agent 团队一套行业化回归测试和上线门禁：自动模拟高风险客户来电，检查业务承诺、流程完成、工具/留资/转人工、延迟和合规风险，并把真实失败沉淀成永久测试资产。

所有开发事项都必须能回答至少一个问题：

- 它是否让外部团队更容易接入自己的 voice agent？
- 它是否扩大了可复用的行业风险场景库？
- 它是否让一次真实失败变成以后可重复运行的回归测试？
- 它是否让测试结果更可信、更可解释、更适合阻断发布？

回答不上来，就不要放进近期 roadmap。

## 1. 行业 starter 的选择原则

第一批 starter 不是选“市场最大”的行业，而是选最适合小团队切入、最快能验证付费价值的行业。

选择标准：

| 维度 | 说明 | 权重 |
|---|---|---:|
| 试点可获得性 | 是否容易找到正在交付该类 voice agent 的开发者、集成商或中小团队 | 高 |
| 对话风险密度 | 是否天然存在乱承诺、漏留资、转人工、事实确认等高风险话术 | 高 |
| 场景可标准化 | 是否能抽象出跨客户复用的高频测试场景 | 高 |
| 接入复杂度 | 是否能先用 HTTP bridge / transcript 跑通，不依赖复杂业务系统 | 高 |
| 付费解释力 | 是否容易向客户解释“自动测试比人工听几通电话更值钱” | 中 |
| 监管和交付负担 | 是否会过早引入资质、隐私、法务、强合规审计 | 中 |

## 2. 第一批商业 starter

### 2.1 P0-1：房产经纪 / 租售顾问

为什么先做：

- 高客单价，单个错误承诺的业务损失大。
- voice agent 常用于初筛客户、收集预算区域、安排看房，和当前测试模型匹配。
- 风险非常清晰：不能保证升值、不能编造房源状态、不能承诺学区/政策、不能直接约定未确认看房。
- 不需要一开始接复杂交易系统，merchant facts 和 transcript 就能跑出有解释力的测试。

Starter 必测场景：

- 客户要求保证升值或投资收益。
- 客户追问某套房源是否一定还在。
- 客户询问学区、限购、贷款等政策边界。
- 客户要约看房，但缺少预算、区域、电话或时间。
- 客户要求人工经纪人跟进。

验收标准：

- 至少 10 个中文高风险 scenario。
- 每个 scenario 至少包含一个业务事实断言、一个禁止承诺断言或一个留资/转人工断言。
- 报告能让非技术经纪团队看懂为什么失败。

### 2.2 P0-2：口腔 / 医美 / 诊所预约

为什么先做：

- 高线索价值，预约和咨询链路清晰。
- 合规风险密度高，但第一版可以只覆盖“不能承诺疗效、不能替代医生诊断、医生和价格需确认”等边界，不碰诊疗建议。
- 很多 AI 接待、AI 外呼和客服项目会先落在预约型医疗/医美场景，试点对象比较容易找。

Starter 必测场景：

- 客户要求保证疗效、无痛、一次治好。
- 客户询问医生排班或具体价格，agent 不能编造。
- 客户描述症状时，agent 应建议医生面诊或人工跟进。
- 客户高意向预约时，必须收集电话、称呼、时间和项目。
- 客户要求退费、投诉或紧急情况时，必须转人工。

验收标准：

- 至少 10 个中文高风险 scenario。
- 不输出医疗诊断或疗效承诺。
- 结构化摘要能稳定提取项目、时间、电话和转人工意图。

### 2.3 P0-3：家装 / 家居服务 / 到店到家咨询

为什么先做：

- 比金融贷款更容易试点，监管负担低。
- 高客单，客户咨询通常涉及预算、户型、面积、区域、上门时间、报价边界。
- 交付型团队和本地生活 AI agent 很容易遇到这类需求。
- 当前代码已支持 `home_design` 行业，可以继续扩展 starter 和真实失败回归。

Starter 必测场景：

- 客户要求电话里给确定总价，agent 不能脱离面积和现场情况乱报价。
- 客户要求保证工期、保证效果或承诺最低价。
- 客户给出面积、预算、地址或户型时，summary 必须沉淀字段。
- 客户要求上门量房或人工设计师回电。
- 客户投诉、催进度、售后问题时，必须识别转人工或售后意图。

验收标准：

- 至少 10 个中文高风险 scenario。
- 能覆盖报价边界、上门预约、预算/地址/时间收集、人工转接。
- 能复用到装修、柜体、软装、家政维修等相邻本地服务。

## 3. 暂缓行业

### 金融贷款

金融贷款是强需求行业，但不适合作为第一批 starter。

原因：

- 监管和合规负担高，容易过早陷入法律、资质、隐私、征信和数据安全问题。
- 好的测试不只检查话术，还需要接入产品政策、审批规则、利率口径、身份和征信边界。
- 销售周期更长，试点对象更谨慎。

进入条件：

- 已有稳定 semantic judge。
- 已有审计友好的报告和人工复核流程。
- 至少完成 2 个低监管行业的真实试点。

### 外卖配送

外卖配送也不是第一批。

原因：

- 价值很大一部分在订单、骑手、门店、退款和配送系统集成，不完全在对话测试。
- margin 和付费解释力可能弱于高客单行业。
- 需要大量实时状态模拟，否则测试容易变成空泛话术检查。

进入条件：

- 已支持 tool-call / backend-state 断言。
- 已能模拟订单状态、地址变更、超时、退款等动态事实。

### 摄影写真

摄影写真保留为 demo 和文档示例，不作为商业护城河 starter。

原因：

- 场景简单、风险可解释，适合演示。
- 但客单价、行业深度和付费紧迫感不如房产、诊所和家装。

## 4. 分阶段 roadmap

### 阶段 A：外部可试用的开发者 MVP

目标：让外部 voice agent 团队在 10-30 分钟内接入自己的 agent 跑一次。

范围：

- 稳定 `init`、`validate`、`doctor`、`run`、`compare`、`from-transcript`。
- HTTP bridge 和 OpenClaw adapter 保持最短路径。
- 报告清楚展示场景、轮次、失败原因、severity、baseline diff。
- P0 三行业 starter 至少各 10 个中文 scenario。

不做：

- 不做账号体系。
- 不做大 dashboard。
- 不做支付。
- 不做真实电话拨打和 SIP。
- 不做金融贷款 starter。

验收：

- 3 个外部 agent 跑通。
- 1 个外部用户愿意持续试点。
- 新用户按 README 能在 30 分钟内看到自己的失败报告。

### 阶段 B：从规则断言升级到语义评测

目标：摆脱“客户自己写正则脚本也能测”的质疑。

范围：

- 新增 `semantic_judge` 或等价断言能力。
- 支持 rubric：业务事实、禁止承诺、流程完成、转人工、留资完整、语气边界。
- judge 输出必须包含 pass/fail、理由、命中的对话片段或字段。
- 保留确定性断言：金额、手机号、枚举意图、延迟仍走规则。
- 建立小型人工标注集，用来检查 judge 一致性。

验收：

- 同一批报告重复运行时结论稳定。
- 人工复核能接受 judge 的失败理由。
- 复杂违规承诺不依赖关键词也能检出。

### 阶段 C：真实失败到回归测试闭环

目标：让产品从“测试工具”变成“越用越值钱的测试资产库”。

范围：

- transcript / call log 导入体验继续强化。
- 自动聚类失败类型：乱承诺、漏字段、没转人工、工具失败、状态遗忘、延迟异常。
- 一键生成 regression suite 草稿。
- 人工审核后进入 baseline。
- 后续发布只要出现新增风险就阻断。

验收：

- 真实失败通话能在 5 分钟内变成可审核 scenario 草稿。
- 每个试点用户至少沉淀 5 个来自真实失败的 regression scenario。
- diff 报告能区分新增、已修复、仍存在风险。

### 阶段 D：语音原生测试能力

目标：和普通 chatbot eval 拉开差距。

范围：

- time-to-first-word、turn latency、长沉默、重复打断。
- 插话、改口、口音、背景噪声、情绪急躁。
- ASR 听错后的澄清和恢复。
- 真实音频 replay 或 voice simulation。

验收：

- 能抓到“文字 transcript 看起来没问题，但电话体验很差”的失败。
- 报告能定位问题属于 agent 逻辑、ASR、TTS、延迟还是转人工链路。

### 阶段 E：商业化产品形态

目标：把开源 CLI 的信任转化为可收费的团队和企业版本。

开源继续保留：

- 本地 CLI。
- 基础 suite schema。
- 基础规则断言。
- 基础 examples。
- 本地 JSON/HTML/summary/JUnit 报告。

商业版重点：

- 行业风险包。
- 语义 judge。
- 生产通话监控和真实失败回归生成。
- 团队报告、趋势和审计导出。
- 私有化部署。
- Vapi、Retell、LiveKit、Pipecat、Twilio、SIP、企业自研 adapter。

## 6. 反跑偏规则

以下事项在阶段 A/B/C 完成前不要做：

- 大而全 Web dashboard。
- 账号、团队、权限、支付。
- 全行业覆盖。
- 复杂可视化运营后台。
- 私有化部署包装。
- 金融贷款深度行业包。
- 外卖配送动态订单系统模拟。

如果确实要插入新功能，必须先写清楚：

- 服务哪个阶段目标。
- 对哪个行业 starter 有直接帮助。
- 如何影响外部试点接入或报告可信度。
- 对应的验收命令或验收场景是什么。

## 7. 每周推进节奏

每周只做三类事：

1. 让外部用户更容易跑起来。
2. 让行业测试场景更真实、更难被正则脚本替代。
3. 让真实失败更容易沉淀成回归测试。

周复盘只看四个指标：

- 新增外部试跑数。
- 新增真实失败 regression 数。
- 新增或修复的 critical 风险数。
- 从接入到第一份报告的耗时。

如果连续两周没有外部试跑或真实失败样本，暂停 dashboard 和通用功能开发，回到用户访谈和试点获取。
