# 小红书商家 AI 语音接待员产品设计

日期：2026-05-03  
状态：设计评审稿  
目标阶段：可商用 MVP

## 1. 背景

团队已有实时语音 AI、RAG、工具调用、通话控制、转人工、录音和摘要等工程经验，并计划借助 OpenClaw 的 Agent 自动化和现成语音集成，减少自研 ASR、LLM、TTS 编排成本。

本设计不把 FreeSWITCH 作为第一阶段必选项。FreeSWITCH 是后续电话/PBX/私有化客户的接入适配器之一。第一阶段核心是做出能推广、能试用、能收费的垂直产品。

## 2. 产品定位

项目代号为“小红书商家 AI 语音接待员”。正式名称在首批商家试用后再定。

一句话定位：

> 帮小红书、抖音、朋友圈来的咨询客户，用 AI 语音快速接待、问清需求、完成预约或留资，并把摘要推给老板。

产品不对外宣传为“AI 语音平台”或“智能客服系统”。早期市场表达必须窄，主打“内容平台线索转预约”。

## 3. 目标客户

第一批客户选择内容获客强、咨询重复、客单价较高、合规压力较低的预约型商家：

- 摄影写真、婚礼摄影、亲子摄影。
- 家装设计、软装咨询、局改装修。
- 健身私教、瑜伽普拉提、运动康复中的非医疗咨询。
- 宠物服务、宠物摄影、宠物寄养。

第一阶段优先做两个行业模板：

1. 摄影写真/婚礼摄影。
2. 家装设计/软装咨询。

原因：

- 小红书内容获客明显，适合做公开演示。
- 咨询问题复杂，AI 接待价值比简单 FAQ 更明显。
- 客单价能覆盖月费。
- 医疗、金融、法律等强监管风险较低。

## 4. 不做什么

第一阶段不做以下范围：

- 不做冷外呼营销。
- 不做全渠道客服中心。
- 不做复杂工单系统。
- 不做私有化部署。
- 不做 FreeSWITCH 生产级接入。
- 不自研完整 ASR、LLM、TTS pipeline。
- 不承诺替代人工成交，只负责接待、筛选、预约、留资和通知。

## 5. 核心用户故事

### 5.1 商家

商家希望把小红书笔记、主页、私信、评论或朋友圈流量导向一个 AI 接待入口。客户进来后，AI 能按行业话术问清需求，回答基础问题，收集联系方式，推动预约，并把结构化摘要发给老板。

### 5.2 客户

客户不想填复杂表单，也不想等商家回复。客户希望像打电话或发语音一样快速问清价格、档期、地址、风格、服务范围，并得到下一步预约安排。

### 5.3 团队

团队希望用最短路径验证商家愿意为“线索转预约”付费，而不是先投入数月构建通用平台。

## 6. MVP 功能

### 6.1 商家配置

- 商家基础信息：店名、地址、服务区域、营业时间、联系方式。
- 行业模板：摄影写真、家装设计。
- 服务套餐：名称、价格区间、包含内容、适合人群。
- 常见问题：价格、档期、拍摄地点、装修预算、施工周期等。
- 预约规则：可预约时间、预约前置问题、是否需要人工确认。
- 人工接管信息：老板微信、手机号、飞书 webhook 或企微 webhook。

### 6.2 客户语音入口

第一版实现网页语音入口：

- 商家获得一个可分享链接或二维码。
- 客户打开页面后点击语音按钮开始咨询。
- 页面展示商家名称、服务类型和简短提示。
- 语音对话结束后，页面展示预约或留资结果。

电话入口作为第二阶段：

- 优先接 Twilio、Telnyx、Plivo 或国内云通信回拨。
- FreeSWITCH 放到企业客户或私有化部署阶段。

### 6.3 AI 接待

AI 接待必须完成以下动作：

- 简短开场，说明自己是商家的 AI 接待助手。
- 识别客户意图：咨询价格、咨询档期、了解服务、预约、投诉或转人工。
- 按行业模板问关键问题。
- 根据商家配置回答基础问题。
- 不确定时明确说“需要商家确认”，不得编造价格、档期或承诺。
- 收集客户姓名、联系方式、需求、预算、时间偏好。
- 生成预约意向或留资摘要。
- 必要时转人工或提示老板稍后联系。

### 6.4 老板通知

每次有效咨询结束后，系统向老板发送结构化摘要：

- 客户姓名和联系方式。
- 客户来源：小红书、抖音、朋友圈、官网或未知。
- 意向行业和服务项目。
- 预算范围。
- 时间偏好。
- 客户关心的问题。
- AI 判断的意向等级：高、中、低。
- 建议下一步动作。
- 对话转写链接。

### 6.5 后台

第一版后台保持轻量：

- 商家列表。
- 每个商家的配置表单。
- 咨询记录列表。
- 单次咨询详情。
- 行业模板编辑。
- 通知配置。

第一版不做在线支付、发票、复杂权限和商家自助注册。商家由运营人员手动开通，套餐状态也由运营人员手动调整。

## 7. 技术架构

### 7.1 总体架构

```text
内容平台线索
  -> 商家专属咨询页
  -> WebRTC/网页语音入口
  -> 语音 Agent Provider
  -> OpenClaw Agent 自动化
  -> 行业知识和商家配置
  -> 预约/留资/通知工具
  -> 咨询摘要和后台
```

### 7.2 组件边界

#### Voice Entry Adapter

职责：

- 管理网页语音会话。
- 处理麦克风授权、连接状态、断线重连。
- 将音频会话交给现成语音 Agent provider。
- 记录会话 ID、商家 ID、来源参数。

第一版只实现 WebRTC/网页语音。电话、FreeSWITCH、SIP、云通信回拨都作为后续 adapter。

#### Voice Agent Provider

职责：

- 承担低延迟语音能力。
- 处理 ASR、TTS、打断、语音流式响应。
- 向 OpenClaw Agent 发送结构化对话事件。

选择原则：

- 优先使用 OpenClaw 已支持或容易接入的语音方案。
- 不在第一版自研 ASR、LLM、TTS 编排。
- provider 必须支持工具调用或可通过中间服务触发 OpenClaw 自动化。

#### OpenClaw Agent Layer

职责：

- 管理行业接待流程。
- 调用知识库、预约、通知、CRM 工具。
- 输出适合语音播报的回复。
- 生成结构化 lead summary。

这里是业务自动化核心，不是媒体处理核心。

#### Merchant Profile Service

职责：

- 保存商家配置。
- 根据商家链接解析商家 ID。
- 提供套餐、FAQ、营业时间、预约规则。
- 为 OpenClaw Agent 生成上下文。

#### Lead Workflow Service

职责：

- 创建咨询记录。
- 保存对话摘要和转写。
- 判断是否有效线索。
- 发送老板通知。
- 后续接入飞书、企微、表格、CRM、日历。

#### Admin Console

职责：

- 配置商家信息。
- 查看咨询记录。
- 编辑模板。
- 配置通知渠道。

### 7.3 第一版技术选型

默认技术选型如下，除非实现阶段发现 OpenClaw 集成要求必须调整：

- 前端和后台：Next.js + TypeScript。
- 后端 API：Next.js Route Handlers 或轻量 Node.js 服务。
- 数据库：PostgreSQL。
- 缓存和任务队列：Redis。
- Agent 自动化：OpenClaw。
- 语音入口：网页 WebRTC 或浏览器麦克风流。
- 语音 provider：优先选择能和 OpenClaw 工具调用闭环最短的 provider。
- 通知：第一版优先支持飞书 webhook；微信和企微作为人工转发或第二阶段集成。

选择这个栈的原因是第一版需要快速做出演示、后台、商家链接和通知闭环。现有 Python 语音经验可以沉淀为后续 provider adapter，但不作为第一版主路径。

## 8. 数据流

### 8.1 初次配置

```text
运营人员创建商家
  -> 选择行业模板
  -> 填写商家资料、套餐、FAQ、预约规则
  -> 生成商家专属链接和二维码
  -> 商家放到小红书主页、私信自动回复、笔记评论或朋友圈
```

### 8.2 客户咨询

```text
客户打开链接
  -> 点击语音咨询
  -> Voice Entry Adapter 创建会话
  -> Voice Agent Provider 开始实时语音
  -> OpenClaw Agent 根据商家配置接待
  -> 工具收集预约/留资信息
  -> Lead Workflow Service 保存摘要
  -> 通知老板跟进
```

### 8.3 人工接管

```text
客户要求人工或 AI 判断高价值
  -> OpenClaw Agent 触发 handoff 工具
  -> Lead Workflow Service 发送高优先级通知
  -> 页面提示“商家会尽快联系”
  -> 第二阶段可支持实时电话转接
```

## 9. 错误处理

### 9.1 麦克风失败

- 页面提示客户改用文字留言或电话联系。
- 系统记录失败原因为 `microphone_denied` 或 `microphone_unavailable`。

### 9.2 语音 provider 失败

- 页面提示“当前语音接待繁忙，请留下联系方式”。
- 自动切换到文字表单。
- 通知运营人员排查。

### 9.3 OpenClaw 工具失败

- AI 不播报内部错误。
- 对客户说“这项信息需要商家确认，我会记录下来让商家联系你”。
- 摘要中标记 `tool_error=true`。

### 9.4 通知失败

- 咨询记录必须已落库。
- 通知失败进入重试队列。
- 后台展示未通知状态。

### 9.5 幻觉和错误承诺

- 商家配置中没有的信息不得推测。
- 价格、档期、优惠、合同、医疗效果、法律承诺等敏感内容必须使用保守话术。
- 不确定时转为“待商家确认”。

## 10. 商业化设计

### 10.1 定价

第一阶段建议三档：

- 试用版：7 天，20 次有效咨询，适合获客。
- 基础版：299 元/月，包含一个商家、一个行业模板、200 次有效咨询。
- 增长版：699 元/月，包含一个商家、两个行业模板、800 次有效咨询、优先通知和模板定制。

不建议第一版按 token 或语音分钟单独售卖。商家更容易理解“每月帮你接待多少咨询”。

有效咨询定义为客户至少完成一轮明确业务问题，或留下联系方式，或触发老板通知。无麦克风授权、页面打开后立即退出、系统错误不计入有效咨询。

### 10.2 付费触发点

商家愿意付费的证据：

- AI 接待带来真实预约或高意向留资。
- 老板认为摘要有用，能减少重复沟通。
- 商家愿意把链接放到小红书主页、私信或朋友圈。
- 试用结束后愿意继续使用。

### 10.3 首批交付方式

前 10 个客户采用半人工交付：

- 团队帮商家整理 FAQ 和套餐。
- 团队手动优化行业模板。
- 团队每天复盘对话失败原因。

目标不是自动化 onboarding，而是尽快拿到真实对话和付费反馈。

## 11. 增长实验

### 11.1 内容渠道

优先渠道：

- 小红书。
- 抖音。
- 微信朋友圈。
- 即刻、掘金、V2EX 等开发者社区用于招募合作商家或代理。

### 11.2 内容选题

前 30 天发布以下内容：

- “我给摄影工作室做了一个 AI 接待员”。
- “客户问价格、档期、风格，AI 如何帮老板留资”。
- “小红书来的客户不想等回复，可以这样自动接待”。
- “AI 接待员和普通表单的区别”。
- “老板收到的客户摘要长什么样”。

### 11.3 验证指标

30 天验证目标：

- 发布 30 条内容。
- 触达 100 个目标商家。
- 拿到 20 个试用意向。
- 完成 5 个商家配置。
- 产生 100 次真实或半真实咨询。
- 拿到 2 个付费或明确续试意向。

未达到 2 个付费或续试意向时，不扩大开发范围，只调整目标行业、话术和获客内容。

## 12. 竞品和护城河

### 12.1 竞品判断

泛 AI 电话客服和语音 Agent 竞品很多，国内外都有成熟产品。直接进入“AI 语音平台”会陷入红海。

本产品避开泛平台竞争，聚焦“内容平台线索转预约”。竞品主要来自：

- 平台私信和线索工具。
- 有赞、微盟、SCRM 工具。
- 普通客服外包。
- 通用 AI 客服和电话机器人。

### 12.2 护城河

早期技术护城河弱。可形成的护城河是：

- 行业模板和真实对话数据。
- 线索转化评测集。
- 商家工作流集成。
- 内容获客和 demo 生产能力。
- 低成本、快速配置、快速交付能力。

## 13. 测试策略

### 13.1 单元测试

- 商家配置解析。
- 行业模板选择。
- 摘要结构化。
- 通知 payload 生成。
- 有效线索判断。

### 13.2 集成测试

- 创建商家后生成咨询链接。
- 客户完成一次语音咨询后生成 lead。
- 通知失败后进入重试队列。
- OpenClaw 工具失败时使用保守话术。

### 13.3 对话评测

每个行业维护最少 50 条测试问题：

- 价格咨询。
- 档期咨询。
- 地址和服务范围。
- 预算和需求收集。
- 要求人工。
- 敏感承诺。
- 模糊问题。

每次模板变更后运行评测，确认：

- 没有编造价格。
- 没有漏收联系方式。
- 没有错误承诺。
- 能正确生成摘要。

### 13.4 人工验收

每个商家上线前人工跑 10 轮模拟咨询：

- 3 轮普通咨询。
- 3 轮价格和档期咨询。
- 2 轮高意向预约。
- 1 轮要求人工。
- 1 轮语音失败或中断。

## 14. 里程碑

### M0：设计和计划

- 完成本设计文档。
- 完成实现计划。
- 明确第一版技术选型。

### M1：可演示原型

- 支持一个商家配置。
- 支持网页语音咨询。
- 支持 OpenClaw Agent 接待。
- 支持生成摘要。
- 支持发送老板通知。

### M2：可试用 MVP

- 支持两个行业模板。
- 支持多商家。
- 支持咨询记录后台。
- 支持通知失败重试。
- 支持基本对话评测。

### M3：可收费版本

- 支持试用额度。
- 支持套餐和用量统计。
- 支持运营人员手动开通和续费记录。
- 支持商家二维码。
- 支持基础运营看板。
- 支持至少 5 个真实商家试用。

### M4：电话扩展

- 接入一个电话 provider。
- 支持客户点击后电话回拨或商家号码呼入。
- 根据真实需求评估是否接 FreeSWITCH。

## 15. 成功标准

产品进入“可商用 MVP”的标准：

- 商家能在 30 分钟内完成基础配置。
- 客户能通过链接完成一次语音咨询。
- 系统能生成结构化 lead summary。
- 老板能在微信、飞书或企微收到通知。
- 后台能查看咨询记录。
- 两个行业模板通过基础对话评测。
- 真实商家试用中至少产生 2 个愿意继续使用或付费的客户。

## 16. 后续决策点

只有满足以下条件时才进入电话和 FreeSWITCH 深水区：

- 已有真实商家要求电话入口。
- 网页语音已验证能产生有效线索。
- 至少 2 个付费客户愿意为电话入口支付额外费用。
- 团队已沉淀稳定的行业模板和摘要工作流。

否则继续优化网页语音入口、行业模板和获客内容。
